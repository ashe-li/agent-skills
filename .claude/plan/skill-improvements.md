## Overview

根據 dry-run 驗收結果，針對 `/update`、`/design`、`/assist` 三個 skill 進行 4 項改善。每項改善都用 BEFORE/AFTER 標示具體修改。

## 改善 1: /update — 無變更 edge case

**問題：** git diff 為空時，流程沒有定義該怎麼辦。doc-updater 會在沒有目標的情況下空轉。

**檔案：** `update/SKILL.md`

### BEFORE (L21-31)

```markdown
**掃描範圍：**

1. 執行 `git diff --name-only HEAD` 取得本次變更的檔案清單
2. 若無未 commit 變更，使用 `git diff --name-only HEAD~3..HEAD` 取得近期 commit 涉及的檔案
3. 根據變更內容更新：
   - `docs/` 目錄下的相關文件
   - `docs/CODEMAPS/` 目錄下的架構圖
   - `README.md`（如果功能或用法有變動）
   - 其他受影響的文件檔案

**交接資訊：** 記錄更新了哪些文件檔案、變更摘要，作為下一步 code-reviewer 的輸入。
```

### AFTER

```markdown
**掃描範圍：**

1. 執行 `git diff --name-only HEAD` 取得本次變更的檔案清單
2. 若無未 commit 變更，使用 `git diff --name-only HEAD~3..HEAD` 取得近期 commit 涉及的檔案
3. 若仍無任何變更，使用 AskUserQuestion 詢問：
   - **指定檔案** — 手動指定要檢查的文件
   - **全面掃描** — 掃描所有文件是否有過時內容
   - **只做 learn-eval** — 跳過 Step 1-2，直接進入 Step 3 提取 patterns
   - **結束** — 結束 /update 流程
4. 根據變更內容更新：
   - `docs/` 目錄下的相關文件
   - `docs/CODEMAPS/` 目錄下的架構圖
   - `README.md`（如果功能或用法有變動）
   - 其他受影響的文件檔案

**交接資訊：** 記錄更新了哪些文件檔案、變更摘要，作為下一步 code-reviewer 的輸入。
```

**差異：** 在第 2 和第 4 點之間插入第 3 點 — 當兩種 git diff 都為空時，詢問使用者要怎麼做，而不是讓 doc-updater 空轉。

## 改善 2: /design — 精簡流程 + 複雜度評估

**問題：**
- Step 4 用 doc-updater「預估文件影響」— 但 doc-updater 是「更新文件」的工具（model: haiku），不是「預估」的
- Step 5 用 code-reviewer「審查計畫」— 但 code-reviewer 第一步就跑 `git diff`，對文字計畫完全無效
- 不管需求多簡單，都跑完整 6 步流程

**檔案：** `design/SKILL.md`

### ECC 工具適配分析

先確認哪些 ECC 工具適合「計畫階段」vs「實作階段」：

| ECC 工具 | 適用階段 | 理由 |
|----------|---------|------|
| **planner** (agent, opus) | 規劃 | 核心職責就是「分析需求、建立計畫、識別風險」 |
| **architect** (agent, opus) | 規劃 | 核心職責是「評估架構權衡、推薦模式」，文件影響是架構分析的延伸 |
| **doc-updater** (agent, haiku) | 實作後 | 核心職責是「codemap 生成、文件更新」，需要有實際程式碼才能運作 |
| **code-reviewer** (agent, sonnet) | 實作後 | 核心職責是「審查程式碼品質」，第一步是 `git diff` |
| **/update-docs** (command) | 實作後 | 從 package.json、.env、route 檔案同步文件 |
| **/update-codemaps** (command) | 實作後 | 分析程式碼結構生成架構文件 |
| **/eval** (command) | 規劃或實作後 | 定義評估標準、執行評估、生成報告 |

結論：規劃階段只該用 planner + architect。doc-updater、code-reviewer、update-docs、update-codemaps 都是實作後的工具，不該出現在規劃流程中。

### BEFORE（完整 6 步流程）

```
Step 1: 盤點 ECC 資源
Step 2: planner — 建立實作計畫
Step 3: architect — 架構設計審查
Step 4: doc-updater — 預估文件影響     ← 實作後工具，不該在規劃階段
Step 5: code-reviewer — 審查計畫品質   ← 實作後工具，不該在規劃階段
Step 6: 輸出 plan.md
```

### AFTER（精簡為 5 步 + 複雜度分流）

```
Step 1: 盤點 ECC 資源
Step 2: 複雜度評估（新增）— 決定走完整流程還是快速路徑
Step 3: planner — 建立實作計畫
Step 4: architect — 架構設計審查 + 文件影響評估（architect 的架構分析自然延伸）
Step 5: 計畫品質自審 + 輸出 plan.md
```

移除的工具去哪裡了：
- **doc-updater** → 不在 /design 流程中。plan.md 的實作步驟會建議「實作完成後執行 `/update`（用 doc-updater + update-docs + update-codemaps 同步文件）」
- **code-reviewer** → 不在 /design 流程中。plan.md 的實作步驟會建議「程式碼寫完後用 code-reviewer 審查」

### 具體修改 A: architect 承接文件影響評估

**Step 4 architect 的審查重點改為：**

```markdown
**審查重點：**

- 架構是否合理、可擴展
- 是否有更簡單的替代方案
- 與現有程式碼的相容性
- 效能和安全性考量
- ECC 資源的選用是否恰當
- 文件影響評估：實作後需要新增或更新哪些文件（README、API docs、CODEMAPS），列入 plan.md 待辦
```

與現在的差異：多了最後一行「文件影響評估」。為什麼 architect 適合做這件事：
- architect 用 opus model，有全局推理能力
- 核心職責已包含「Component responsibilities」和「Integration points」分析
- 評估「哪些文件會受影響」是架構分析的自然延伸

### 具體修改 B: 計畫品質改為內聯檢查表

**原 Step 5（code-reviewer 審查計畫）改為：**

```markdown
## Step 5: 輸出 plan.md

將最終計畫寫入檔案前，先對照以下檢查清單確認計畫品質：

| 維度 | 通過標準 |
|------|---------|
| 完整性 | 每個需求都有對應的實作步驟 |
| 可執行性 | 每個步驟有明確的檔案路徑和具體動作 |
| 依賴正確性 | 步驟間依賴無環且順序合理 |
| 粒度適當 | 每個步驟 1-3 個具體動作 |
| ECC 資源合理 | 每個 agent/skill 在其設計用途內使用 |
| 驗收可測 | 每個驗收標準都可客觀驗證 |
| 實作後工具 | plan.md 的步驟中包含實作後的品質保障（code-reviewer、/update、/verify 等） |

如果發現問題，直接修正計畫內容。

**輸出路徑：** ...（以下 plan.md 格式不變）
```

與現在的差異：不再呼叫 code-reviewer agent，改為 7 項檢查表自審。最後一項「實作後工具」確保 plan.md 的步驟會在正確時機引入 code-reviewer、doc-updater 等工具。

### 具體修改 C: 新增 Step 2 複雜度評估

在 Step 1 之後插入：

```markdown
## Step 2: 複雜度評估

根據需求規模，選擇執行路徑：

| 複雜度 | 判斷標準 | 執行路徑 |
|--------|---------|---------|
| **低** | 單一檔案、不涉及架構變更（例：更新 README、加 badge、改設定） | 跳過 Step 4（architect），直接 Step 3 → Step 5 |
| **中等以上** | 跨檔案或有架構影響 | 完整流程 Step 3 → Step 4 → Step 5 |

如果需求過於模糊無法判斷複雜度，使用 AskUserQuestion 請使用者補充具體資訊。

> 無論走哪條路徑，都會輸出 plan.md。
```

### 具體修改 D: plan.md 模板確保實作後工具出現在正確位置

plan.md 模板中的 Implementation Steps 區塊要求包含實作後的品質步驟：

```markdown
## Implementation Steps

### Phase 1: [實作階段]
- [ ] Step 1: ...

### Phase 2: [品質保障]
- [ ] 執行 code-reviewer 審查程式碼品質
- [ ] 執行 security-reviewer 檢查安全性
- [ ] 執行 /verify 進行全面驗證

### Phase 3: [文件同步]
- [ ] 執行 /update 更新知識庫（doc-updater + learn-eval）
- [ ] 或手動執行 /update-docs + /update-codemaps

## Architecture Notes
<!-- architect 的審查結果（含文件影響評估）-->

## Review Notes
<!-- 計畫品質自審的結果 -->
```

這樣 doc-updater 和 code-reviewer 仍然會出現在 plan.md 中，只是在正確的時機（實作後）而非規劃階段。

## 改善 3: /assist — 衝突優先順序

**問題：** 同時偵測到多個情境時（例如使用者要求新功能，但 build 失敗），沒有定義先處理哪個。

**檔案：** `assist/SKILL.md`

### BEFORE (L89-94)

```markdown
### 路由判斷邏輯

1. **優先使用 `$ARGUMENTS`**：如果使用者有明確指令，以指令為準
2. **其次分析 git 狀態**：未 commit 變更 → review pipeline；build 失敗 → build-fix pipeline
3. **再看對話脈絡**：從對話中推斷使用者的意圖
4. **無法判斷時詢問**：使用 AskUserQuestion 提供 2-4 個建議選項
```

### AFTER

```markdown
### 路由判斷邏輯

1. **優先使用 `$ARGUMENTS`**：如果使用者有明確指令，以指令為準
2. **其次分析 git 狀態**：未 commit 變更 → review pipeline；build 失敗 → build-fix pipeline
3. **再看對話脈絡**：從對話中推斷使用者的意圖
4. **無法判斷時詢問**：使用 AskUserQuestion 提供 2-4 個建議選項

### 多情境衝突處理

當同時偵測到多個情境時，依以下優先順序決定先處理哪個：

1. Build 失敗 — 必須先修復，否則其他工作無法驗證
2. 安全問題 — 不能延後
3. 使用者明確指令（$ARGUMENTS）— 使用者意圖優先
4. Bug 修復 — 修復比新功能重要
5. 新功能開發 — 標準優先級
6. 重構/清理 — 較低優先級
7. 文件更新 — 可延後或與其他工作並行

若偵測到的情境與 `$ARGUMENTS` 衝突（例如使用者要 review 但 build 失敗），先說明偵測結果，詢問使用者是否仍要繼續原指令。
```

**差異：** 在原有的 4 點判斷邏輯之後，新增一個「多情境衝突處理」段落，定義 7 層優先順序。

## 改善 4: /assist — Handoff Protocol 加 Status

**問題：** 下一個 agent 無法快速判斷前一步是成功還是失敗，必須從自然語言中推斷。

**檔案：** `assist/SKILL.md`

### BEFORE (L117-134)

```markdown
## HANDOFF: [previous-agent] -> [next-agent]

### Context
<!-- 任務背景和目標 -->

### Findings
<!-- 上一個 agent 的發現和產出 -->

### Files Modified
<!-- 被修改的檔案清單 -->

### Open Questions
<!-- 未解決的問題 -->

### Recommendations
<!-- 給下一個 agent 的建議 -->
```

### AFTER

```markdown
## HANDOFF: [previous-agent] -> [next-agent]

### Status: [COMPLETED | COMPLETED_WITH_ISSUES | FAILED]
<!-- COMPLETED = 正常繼續 | COMPLETED_WITH_ISSUES = 繼續但標記 | FAILED = 暫停詢問使用者 -->

### Context
<!-- 任務背景和目標 -->

### Findings
<!-- 上一個 agent 的發現和產出 -->

### Files Modified
<!-- 被修改的檔案清單 -->

### Open Questions
<!-- 未解決的問題 -->

### Recommendations
<!-- 給下一個 agent 的建議 -->
```

**差異：** 在 HANDOFF 標題之後插入一行 `Status` 欄位和一行註解。其他欄位不變。
