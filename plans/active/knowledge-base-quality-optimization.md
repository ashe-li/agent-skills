# Implementation Plan: Knowledge Base Quality Optimization

> Generated by /design on 2026-03-14
> Plan file: plans/active/knowledge-base-quality-optimization.md
> Status: PENDING APPROVAL

## Overview

知識庫品質下降的根因：`/update` 的 learn-eval 只提取 code patterns，遺漏對話中的決策脈絡；Step 4 只被動報告差異，不主動寫入 MEMORY.md。本計畫透過三個獨立可交付的 Phase 修復：擴展提取範圍、強化主動寫入、建立品質管控。

## 根因對照表

| 根因 | 影響 | 解法 Phase |
|------|------|-----------|
| learn-eval 提取範圍過窄 | 對話中的決策脈絡、trade-offs、放棄方案被遺漏 | Phase 1 |
| `/update` Step 4 被動報告，不主動寫入 | MEMORY.md 空置，專案級決策無持久化 | Phase 1 |
| learned skills 格式不一致 | 部分無 frontmatter、評分格式不統一，難以檢索 | Phase 2 |
| 142 個 learned skills 無品質管控 | 過時/過度特定的 patterns 累積 | Phase 2 |

**已排除的方案（architect 審查後）：**
- `/pr` Context 區塊自動回流知識庫 -- 與 learn-eval 來源重疊，時序衝突
- 新增獨立的「對話 context 提取 agent」 -- 與 learn-eval 重疊，token 成本高
- Learned skills 自動合併/過時偵測 -- 風險過高，需人工判斷

## Implementation Steps

### Phase 1: 強化 `/update` 的知識提取與寫入

**目標：** 讓 `/update` 不只提取 code patterns，也捕獲決策脈絡，並主動寫入 MEMORY.md。

- [ ] Step 1: 擴展 learn-eval 提取範圍
  - File: `update/SKILL.md` Step 3
  - Action: 在「提取範圍」清單新增四類知識
  - 新增項目：
    - 決策脈絡（decision context）-- 為什麼選 A 不選 B，含 trade-offs
    - 放棄的方案（rejected alternatives）-- 嘗試過但放棄的做法及原因
    - 對話中的業界/學術依據 -- RFC、OWASP、論文引用的結論摘要
    - 專案演進記錄 -- 設計決策如何隨時間改變
  - 新增「提取觸發訊號」：對話中出現以下內容時必須提取 →
    - 「考慮過 X 但...」「最終選擇 Y 因為...」「放棄了 Z 方案」
    - 「根據 RFC/OWASP/...」「業界做法是...」
    - 明確的 before/after 比較
  - Risk: 低 -- 只修改提取說明，不改流程

- [ ] Step 2: 新增知識分類指引
  - File: `update/SKILL.md` Step 3
  - Action: 在提取範圍後新增「知識分類」表格
  - 分類原則：

    | 知識類型 | 目標位置 | 判斷標準 |
    |----------|---------|---------|
    | 跨專案可重用 pattern | `~/.claude/skills/learned/` | 換一個專案還有用嗎？ |
    | 專案特定架構決策 | MEMORY.md `project` type | 只對此專案有意義 |
    | 使用者偏好修正 | MEMORY.md `feedback` type | 使用者明確糾正了行為 |
    | 外部系統指標 | MEMORY.md `reference` type | Grafana/Linear/Notion 等連結 |

  - 新增「Decision Context」learned skill 格式：
    ```markdown
    # Decision: [決策標題]
    ## Alternatives Considered
    ## Decision Made
    ## Rationale
    ## Trade-offs Accepted
    ## When to Revisit
    ```
  - Risk: 低

- [ ] Step 3: 強化 Step 4 從被動報告改為主動寫入
  - File: `update/SKILL.md` Step 4
  - Action: 修改交叉比對步驟的行為
  - 現有行為：列出差異 → HITL 確認 →「確認無誤，結束」/「修正後結束」
  - 強化為：
    1. 列出「有做但未記錄的決策/知識」
    2. 為每個遺漏項**起草修正內容**（而非僅告知）
    3. HITL 確認：「確認寫入」/「調整後寫入」/「跳過」
    4. 確認後直接寫入對應知識庫（MEMORY.md 或 learned skill）
  - 新增 MEMORY.md 自動建立邏輯：
    - 定位路徑：`~/.claude/projects/<project-hash>/memory/`
    - project-hash = 專案絕對路徑中 `/` 替換為 `-`，去除開頭 `-`
    - 若目錄/檔案不存在 → 用 AskUserQuestion 確認後建立
  - Risk: 中 -- project-hash 計算需正確；用 Glob 驗證路徑存在

- [ ] Step 4: 新增 learned skill 寫入格式強制規範
  - File: `update/SKILL.md` Step 3
  - Action: 在 learn-eval 說明中加入格式要求
  - 強制 frontmatter：
    ```yaml
    ---
    name: <kebab-case-slug>
    description: "<適用場景一句話>"
    user-invocable: false
    origin: auto-extracted
    ---
    ```
  - 統一品質評分為 5 維度表格格式（廢棄 `specificity=4` 單行格式）
  - Risk: 低

**Phase 1 依賴：** Step 1 → Step 2 → Step 3 → Step 4（線性）

## Architecture Notes

**architect 審查要點摘要：**

1. **知識分類邊界**：learn-eval（跨專案 patterns）vs MEMORY.md（專案特定決策）的邊界需要在 SKILL.md 中明確說明，避免同一知識重複儲存
2. **PR context 回流已排除**：`/pr` 和 `/update` 分析同一對話，回流會造成重複；應讓 `/update` 做完知識提取後，`/pr` 只負責 PR 輸出
3. **Step 4 強化優先於新增步驟**：Step 4 已有交叉比對的框架，從被動改主動比新增步驟更省 token
4. **project-hash 路徑風險**：MEMORY.md 路徑計算需要 Glob 驗證，SKILL.md 中應明確說明計算規則
5. **文件影響**：update/SKILL.md、README.md、CHANGELOG.md 需更新

## Risks & Mitigations

| 風險 | 可能性 | 影響 | 緩解方式 |
|------|--------|------|---------|
| Step 4 主動寫入導致 MEMORY.md 膨脹 | 中 | 中 | 只在「有未記錄的新決策」時觸發；HITL 確認把關 |
| learn-eval 擴展後分類不準確 | 低 | 低 | HITL 確認可攔截；分類表格提供明確判斷標準 |
| `/update /pr` pipeline 串接行為改變 | 低 | 中 | Step 4 寫入不影響 Step 6 串接；去重規則表不受影響 |
| `/curation` 誤刪有效 skill | 低 | 高 | 只自動修正格式；廢棄刪除必須 HITL 確認 |
| project-hash 計算錯誤 | 低 | 中 | Glob 驗證路徑存在；失敗時用 AskUserQuestion 讓使用者指定 |

## Acceptance Criteria

- [ ] `/update` learn-eval 能識別並提取「決策脈絡」類知識（when conversation contains trade-off discussions）
- [ ] `/update` Step 4 偵測到遺漏時，起草修正內容並請使用者確認後寫入

- [ ] `/curation` 能掃描 learned skills 並列出格式問題
- [ ] `/update /pr` pipeline 串接不受影響
- [ ] 修改後的 SKILL.md 檔案不超過 800 行
