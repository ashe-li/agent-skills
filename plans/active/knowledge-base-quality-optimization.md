## Overview

知識庫品質下降的根因：`/update` 的知識提取邏輯只提取 code patterns，遺漏對話中的決策脈絡；交叉比對步驟只被動報告差異，不主動寫入 MEMORY.md。本計畫透過三個獨立可交付的 Phase 修復：擴展提取範圍、強化主動寫入、建立品質管控。

## 根因對照表

| 根因 | 影響 | 解法 Phase |
|------|------|-----------|
| 知識提取範圍過窄 | 對話中的決策脈絡、trade-offs、放棄方案被遺漏 | Phase 1 |
| `/update` 交叉比對步驟被動報告，不主動寫入 | MEMORY.md 空置，專案級決策無持久化 | Phase 1 |
| learned skills 格式不一致 | 部分無 frontmatter、評分格式不統一，難以檢索 | Phase 2 |
| learned skills 無品質管控 | 過時/過度特定的 patterns 累積 | Phase 2 |

**已排除的方案（architect 審查後，2026-03-14 決策，architect agent 已於 2026-04-01 移除）：**
- `/pr` Context 區塊自動回流知識庫 -- 與知識提取來源重疊，時序衝突
- 新增獨立的「對話 context 提取 agent」 -- 與知識提取步驟重疊，token 成本高
- Learned skills 自動合併/過時偵測 -- 風險過高，需人工判斷

## Architecture Notes

**設計決策摘要（原 architect 審查記錄，2026-03-14）：**

1. **知識分類邊界**：知識提取（跨專案 patterns）vs MEMORY.md（專案特定決策）的邊界需要在 SKILL.md 中明確說明，避免同一知識重複儲存
2. **PR context 回流已排除**：`/pr` 和 `/update` 分析同一對話，回流會造成重複；應讓 `/update` 做完知識提取後，`/pr` 只負責 PR 輸出
3. **交叉比對步驟強化優先於新增步驟**：既有交叉比對框架，從被動改主動比新增步驟更省 token
4. **project-hash 路徑風險**：MEMORY.md 路徑計算需要 Glob 驗證，SKILL.md 中應明確說明計算規則
5. **文件影響**：update/SKILL.md、README.md、CHANGELOG.md 需更新
