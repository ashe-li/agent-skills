# Implementation Plan: evidence-check Skill

> Generated by /design on 2026-04-06
> Plan file: plans/active/evidence-check.md
> Status: PENDING APPROVAL

## Context

使用者頻繁使用 prompt 模式「subagent 確認做法是有學術研究和業界標準來支撐最佳實踐，也納入多元參考社群共識/反面意見」來提升決策品質。此模式已部分嵌入 `/design` Step 3/4a 和 `/assist` pipeline，但使用者需要一個**獨立的深度查驗工具**：不依賴既有 pipeline，可對任意技術主張（claim）做原始研究，且接受額外 token 開銷。

本 plan 建立 `/evidence-check` 作為獨立 skill，與 `/design` 的 inline 驗證互補：
- `/design`：計畫建立過程中的 inline 驗證（驗證計畫中已有的引述是否充分）
- `/evidence-check`：事後獨立深度調查（對單一主張做原始多來源研究）

## Overview

`/evidence-check` 接收一個技術主張，啟動 2 個並行 subagent 從 4 維度（D1 學術、D2 業界標準、D3 最佳實踐、D4 社群共識+反面意見）收集證據，偵測跨來源衝突，輸出結構化報告。輸出格式與 `/design` plan 相容，可直接貼入。

## Architecture Decisions

### AD-1: 命名 `evidence-check`
- **選擇**：`evidence-check` — 精確描述功能，無命名衝突
- **排除**：`verify`（與既有 `/verify` 衝突）、`deep-check`（過於模糊）、`research-check`（暗示啟動研究而非查驗）

### AD-2: 2 個並行 `general-purpose` subagent (haiku)
- **選擇**：A 組 D1+D2（學術+業界），B 組 D3+D4（實踐+社群），model="haiku"
- **排除**：4 個 subagent（開銷 4 倍，邊際收益低）、單一循序 subagent（失去並行優勢）

### AD-3: 離散 verdict (AGREE/PARTIAL/CONFLICT/NO-DATA)
- **選擇**：4 個離散狀態
- **排除**：連續信心分數 0.0-1.0（LLM 浮點輸出不穩定，離散更易行動）

### AD-4: 輸出格式對齊 `/design` plan
- **選擇**：表格欄位直接對應 `/design` 的 Industry & Standards Reference 和 Community Consensus 表格
- **排除**：獨立格式（增加貼入時的轉換成本）

### AD-5: Step 0 HITL token 告知
- **選擇**：執行前 AskUserQuestion 告知 ~16,000-30,000 tokens
- **排除**：無 HITL 直接執行（使用者明確要求成本意識）

## Implementation Steps

### Phase 1: Skill 建立

**Impl Step 1: 建立 `evidence-check/SKILL.md` frontmatter 與框架**
- File: `evidence-check/SKILL.md`（新建）
- Frontmatter: `name: evidence-check`, `description: 獨立證據查驗...`, `allowed-tools: Read, Glob, Grep, Write, Agent, AskUserQuestion`, `argument-hint: <技術決策或做法>`, `redundancy-peers: [design, assist]`
- 寫入 Step 0-4 空白骨架

**Impl Step 2: 實作 SKILL Step 0 — HITL token 告知**
- AskUserQuestion：「約 16,000-30,000 tokens，繼續/取消」
- 含差異化說明：此工具為獨立深度調查，不同於 /design 和 /assist 的 inline 驗證

**Impl Step 3: 實作 SKILL Step 1 — 解析主張 + Step 2 — 組裝 context**
- Step 1：接收 $ARGUMENTS，標準化為被驗證主張
- Step 2：依 context 模板組裝 subagent A/B 的 prompt

**Impl Step 3a: 實作 SKILL Step 3 — 並行啟動 subagent**
- 子步驟 1：組裝 subagent context
- 子步驟 2：並行啟動 `Agent(subagent_type="general-purpose", model="haiku")` × 2
- 子步驟 3：合併結果為統一維度表
- Return schema per dimension: `{ status: FOUND|NOT-FOUND|PARTIAL, confidence: HIGH|MEDIUM|LOW, sources: [...], notes: "..." }`

**Impl Step 4: 實作 SKILL Step 4 — 衝突偵測 + verdict**
- 衝突表：`| 衝突點 | D1 | D2 | D3 | D4 | 判定 |`
- Verdict 規則：
  - 4 維 FOUND + AGREE → WELL-SUPPORTED
  - 3 維 FOUND，1 維 PARTIAL/NOT-FOUND → CONDITIONALLY-SUPPORTED
  - 2 維以下 FOUND → WEAKLY-SUPPORTED
  - 有 CONFLICT → CONFLICTED
  - 多數 NOT-FOUND → INSUFFICIENT-EVIDENCE
- 輸出維度摘要表（`| 維度 | 覆蓋狀態 | 信心度 |`）+ `/design` 相容表格
- 末尾加注：「WELL-SUPPORTED 不代表絕對正確，請自行驗證關鍵來源」
- AskUserQuestion：是否寫入 `plans/active/evidence-<slug>.md`

### Phase 2: 品質保障
